ovoscope CI/CD Integration

This page covers how to wire ovoscope end-to-end tests into a repository's CI pipeline using gh-automations reusable workflows, including directory layout, pytest configuration, fixture management, and known CI gotchas.

See also: 451-ovoscope-usage.md for test writing patterns and 450-skill-testing.md for the architecture overview.

Directory Layout

The OVOS workspace convention separates end-to-end tests from unit tests:

my-skill-repo/
├── test/
│   ├── unittests/
│   │   └── test_handlers.py
│   └── end2end/
│       ├── test_intent_match.py
│       ├── test_session_state.py
│       └── fixtures/
│           ├── hello_world_adapt.json
│           └── hello_world_padatious.json
├── pyproject.toml
└── ...

Keeping end2end/ separate allows independent execution — end-to-end tests are slower (they spin up a MiniCroft and train intent models) and require additional dependencies.

pytest Configuration

Using pyproject.toml

[tool.pytest.ini_options]
testpaths = ["test"]

Run subsets independently:

# Fast unit tests only
uv run pytest test/unittests/ -v

# End-to-end tests (requires skill installed + more time)
uv run pytest test/end2end/ -v --timeout=60

End-to-end tests are standard unittest.TestCase subclasses and work with both pytest and python -m unittest discover.

Installing Dependencies in CI

End-to-end tests need both ovoscope and the skill under test installed in the same environment. Add an install step before the test run:

pip install ovoscope
pip install -e .          # registers the skill entry point

When testing multiple skills together:

pip install ovoscope \
    ovos-skill-hello-world \
    ovos-skill-weather

Always verify skill discoverability before running tests:

python -c "
from ovos_plugin_manager.skills import find_skill_plugins
plugins = list(find_skill_plugins())
print('Found skills:', plugins)
assert 'ovos-skill-hello-world.openvoiceos' in plugins
"

GitHub Actions — Integrated with Release Workflow

The standard OVOS pattern is to gate alpha publishing on end-to-end test success. Add an end-to-end job to your release_workflow.yml following the gh-automations conventions:

# .github/workflows/release_workflow.yml
name: Release workflow
on:
  pull_request:
    types: [closed]
    branches: [dev]
  workflow_dispatch:

jobs:
  build_tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: |
          pip install build pytest
          pip install ovoscope
          pip install -e .

      - name: Run unit tests
        run: pytest test/unittests/ -v

      - name: Run end2end tests
        run: pytest test/end2end/ -v --timeout=60

  publish_alpha:
    needs: build_tests
    if: github.event.pull_request.merged == true || github.event_name == 'workflow_dispatch'
    uses: OpenVoiceOS/gh-automations/.github/workflows/publish-alpha.yml@dev
    with:
      propose_release: true
    secrets: inherit

The build_tests job runs before publish_alpha. A failing end-to-end test blocks the release.

Standalone End-to-End Workflow

For repositories that only need end-to-end tests without release automation:

# .github/workflows/end2end.yml
name: End2End Tests
on:
  push:
    branches: [dev, master]
  pull_request:
    branches: [dev]

jobs:
  end2end:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install
        run: |
          pip install ovoscope pytest
          pip install -e .

      - name: Test
        run: pytest test/end2end/ -v --timeout=60

The ovoscope.yml Reusable Workflow

The gh-automations repository provides a dedicated reusable workflow for ovoscope tests:

# .github/workflows/ovoscope.yml
jobs:
  end2end:
    uses: OpenVoiceOS/gh-automations/.github/workflows/ovoscope.yml@dev
    secrets: inherit

All OVOS skill repos and core repos should include this workflow. When adding or updating any gh-automations reference in a repo, also migrate all uses: OpenVoiceOS/gh-automations/...@master lines to @dev in the same PR.

Fixture Management

Fixtures generated by End2EndTest.save() contain the expected message sequence serialised as JSON.

When to commit fixtures:

  • Commit fixtures that test stable, deterministic interactions (e.g., a specific dialog line).
  • Do NOT commit fixtures where the speak utterance varies randomly — either omit the "utterance" key from expected data or write a manual assertion instead.
  • Always generate fixtures with anonymize=True (the default) — this strips real location data from session context.

Selective .gitignore — if you generate fixtures locally but prefer not to commit them, or want to ignore only recorded/non-deterministic artifacts:

# Ignore all fixtures
test/end2end/fixtures/*.json

# Or ignore only recorded artifacts
test/end2end/fixtures/recorded_*.json

Timeout Handling

Padatious intent training can be slow on a cold CI runner. Use a generous --timeout in pytest and pass timeout= to execute():

pytest test/end2end/ -v --timeout=60

test.execute(timeout=30)

For multi-turn tests, increase the timeout proportionally:

test.execute(timeout=60)  # two turns, generous budget

Known CI Gotchas

Skill plugin not found on PATH

Symptom: get_minicroft() hangs or find_skill_plugins() returns an empty list.

Cause: The skill was not installed in editable mode or the entry point was not registered.

Fix: Always install the skill package before ovoscope:

pip install -e .   # registers entry points
pip install ovoscope

Missing .venv in CI

If you use uv locally, your .venv is not present in CI. Use pip directly in CI steps or add a uv pip install step. Do not rely on .venv being pre-activated.

MiniCroft hangs for more than 30 seconds

Padatious training on a cold CI runner can exceed the default timeout. Set a generous --timeout in pytest and confirm the skill entry point is registered correctly.

Flaky tests from session ID collisions

Each test that uses Session("same-id") shares session state with other tests using the same session ID. Use unique session IDs per test class:

import uuid
session = Session(str(uuid.uuid4()))

GUI messages causing assertion failures

By default ignore_gui=True strips GUI namespace messages from the captured sequence. If you see unexpected gui.* messages in failures, check whether a skill emits GUI messages unconditionally and whether expected_messages accounts for them.

Coverage

Run the coverage check after adding end-to-end tests to confirm no regression:

uv run pytest test/ -v --cov=<package_name> --cov-report=term-missing

End-to-end tests contribute to coverage because they exercise real skill code via the full intent pipeline.
